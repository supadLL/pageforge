# PageForge — AI 驱动的可视化前端页面生成器

> 类「墨刀」的桌面端可视化页面搭建工具，融合拖拽编辑、AI 图生页面、AI 代码转换三大能力。

---

## 1. 产品定位

一句话：**画得出来，也写得出来**。

| 维度 | 定位 |
|---|---|
| 用户 | 前端开发者 / 全栈 / 设计师 / 产品 / 学习者 |
| 核心价值 | 拖拽搭原型 + AI 图生代码 + AI 框架转换，一条龙从草图到可工程化代码 |
| 差异点 | 多数低代码工具止步于"生成"，PageForge 强调**生成后可继续精调，并可跨框架转译** |
| 形态 | Electron 桌面应用（跨平台，本地文件友好） |

## 2. 功能模块（细化）

### M1 · 可视化画布编辑器（核心）
- 拖拽：组件库 → 画布
- 选中即编辑：属性面板（样式 / 布局 / 交互 / 数据绑定）
- 图层面板：树形结构，支持拖拽改层级、显隐、锁定、命名
- 工具栏：撤销/重做、对齐、分布、成组/解组、复制/粘贴
- 辅助：标尺、智能对齐线、吸附、网格、断点切换（PC/Pad/Mobile）
- 实时预览（画布内）+ 独立预览窗 + 预览代码视图

### M2 · 组件系统
- 内置：容器/行/列/卡片、文本、标题、按钮、链接、图片、图标、输入框、文本域、选择器、开关、单复选、表格、列表、标签页、模态框、抽屉、轮播、导航栏、页脚、分隔线、徽标、进度条
- 嵌套：容器类组件可承载子组件（树形 DOM 映射）
- 设计令牌（Design Tokens）：颜色板、字号梯度、间距梯度、圆角、阴影、动效；全局主题一处修改处处生效
- 组件 props 化：每个组件有声明式 schema（属性面板据此自动生成）
- 扩展：预留「自定义组件」入口（后续支持导入 React/Vue 组件）

### M3 · 导出与预览
- 导出格式：
  - 单文件 HTML（内联 CSS/JS，便于分享）
  - 分离文件（index.html / styles.css / script.js）
  - 源码视图（直接查看/复制）
- 预览：画布内、独立窗口、设备尺寸模拟、二维码（手机扫码预览本地服务）

### M4 · AI 集成（差异化）
分四层能力：

**M4-1 图生页面（Vision → HTML）**
- 上传截图 / 设计稿 / 手绘草图
- Vision AI 识别布局 + 样式 + 文案 → 生成结构化 HTML+CSS
- 生成结果**加载进画布**（不是只给代码），可继续拖拽精调
- 多张图 → 多区块 → 自动拼装页面

**M4-2 AI 微调（自然语言改页面）**
- 指令式："把所有主按钮改成 #E0321C 圆角 8px"、"顶部加个带搜索框的导航栏"、"这张卡片改成两列"
- 选中范围 AI 改：框选区域后只对该区域生效
- 历史化：AI 改动可撤销，可对比前后

**M4-3 框架转换（HTML → 框架代码）**
- 纯 HTML → **Vue 3 SFC**（template + script setup + scoped style）
- 纯 HTML → **React 函数组件**（JSX + 样式方案可选 CSS Modules / styled / Tailwind）
- 纯 HTML → **uni-app / 微信小程序**（页面 + 组件拆分）
- 纯 HTML → **原生 HTML**（整理、语义化、可读化）
- 可选：Tailwind / 组件库（Element Plus / Ant Design）注入

**M4-4 AI Provider 抽象层**
- 统一接口：`generateFromImage` / `editByPrompt` / `convertFramework`
- 内置适配器：智谱 GLM-4V / GLM-4.5 / OpenAI GPT-4o / Claude（预留）
- 配置：API Key、Base URL、模型选择，存本地（Electron safeStorage 加密）
- 调用走**主进程代理**，密钥永不进入渲染进程 / 不落明文

### M5 · 项目管理
- 多页面项目（一个项目多页面 + 简单路由关系）
- 项目本地保存（.pageforge 工程文件夹：project.json + assets/）
- 模板库（启动页选模板）
- 历史版本（快照）

### M6 · 响应式 & 主题
- 断点：1440 / 1024 / 768 / 375
- 主题：明/暗 + 自定义品牌色
- 全局样式表（自定义 CSS 注入）

## 3. 技术架构

```
┌─────────────────────────────────────────────────────────┐
│                   Electron 桌面应用                       │
│                                                          │
│  ┌───────────────────────────┐   ┌────────────────────┐ │
│  │      渲染进程 (UI)         │   │   主进程 (Main)     │ │
│  │  Vue 3 + Pinia + Vite     │   │  Node.js 服务层     │ │
│  │                           │   │                    │ │
│  │  ┌─────────────────────┐  │   │  · AI Provider 代理 │ │
│  │  │  画布编辑器内核      │  │   │  · 文件读写/导出    │ │
│  │  │  (自研 Schema 驱动)  │◄─┼──►│  · safeStorage 密钥 │ │
│  │  └─────────────────────┘  │IPC│  · 本地预览服务器   │ │
│  │  组件库 / 属性面板 / 图层  │   │  · 项目持久化       │ │
│  │  代码视图 / 预览          │   │                    │ │
│  └───────────────────────────┘   └────────────────────┘ │
└─────────────────────────────────────────────────────────┘
            │                              │
            │  (渲染进程只拿到结构化数据)    │
            ▼                              ▼
      画布数据(JSON)              外部 AI API (主进程代理)
```

### 关键决策
- **编辑器内核自研**（Schema 驱动），而非直接套 GrapesJS：为了完全掌控「图生页面」「框架转译」对同一份数据模型的读写。GrapesJS 数据模型对 AI 不友好且转译困难。
- **Vue 3 渲染层**：Composition API + Pinia，与目标框架之一（Vue）同构，便于复用思路。
- **IPC 边界**：所有文件、网络、密钥操作只在主进程；渲染进程通过类型化 IPC 调用。

## 4. 核心数据模型（Schema）

整个编辑器 + AI + 转译都围绕这棵树。**这是项目的灵魂**。

```jsonc
// Project（工程）
{
  "id": "p_xxx",
  "name": "营销落地页",
  "schemaVersion": 1,
  "tokens": {                 // 设计令牌（全局主题）
    "colors": { "primary": "#E0321C", "text": "#1A1A1A" },
    "spacing": { "unit": 4 },
    "radius": { "md": 8 },
    "fontSize": { "md": 14 }
  },
  "pages": [ /* Page */ ],
  "assets": [ /* 图片等资源 */ ]
}

// Page（页面）
{
  "id": "pg_home",
  "name": "首页",
  "route": "/home",
  "root": { /* Node 根节点 */ }
}

// Node（组件节点 —— 一切皆 Node）
{
  "id": "n_abc",
  "type": "Button",            // 组件类型
  "props": {                   // 组件专属属性（有 schema 约束）
    "text": "立即购买",
    "variant": "primary"
  },
  "style": {                   // 样式（CSS-in-JS 形式，AI 友好）
    "backgroundColor": "$primary",   // $ 引用 token
    "borderRadius": "$radius.md",
    "padding": "12px 24px"
  },
  "children": [ /* Node */ ],  // 子节点（容器类才有）
  "events": [                  // 交互事件
    { "type": "click", "action": { "kind": "navigate", "to": "/pay" } }
  ],
  "responsive": {              // 断点级覆盖
    "mobile": { "style": { "width": "100%" } }
  }
}
```

**为什么这样设计：**
- JSON 可序列化 → AI 直接读写、可 diff、可保存
- `style` 用 CSS-in-JS → AI 生成最自然，渲染到 DOM 也直接
- `tokens` 引用 → 主题一处改，处处变
- `responsive` 断点覆盖 → 多端适配无需多套树
- `props` 有 schema → 属性面板自动生成表单，AI 改组件有据可依

## 5. AI 设计细节

### 5.1 图生页面流程
```
图片 → [Vision Provider] → 结构化描述(JSON: 布局树+样式)
                            │
                            ├── 走严格 JSON Schema 输出 → 直接得到 Node 树 → 入画布
                            └── 兜底：得到 HTML → 解析器转 Node 树 → 入画布
```
- **首选结构化输出**：要求模型直接吐 Node 树 JSON（带 schema 校验），失败再退回 HTML 解析。
- 分块策略：大图先切区块，逐块识别后合并，避免长图信息丢失。

### 5.2 框架转译流程
```
Node 树 → [转译 Prompt] → 目标框架代码
                ↑
        同一份数据，不同模板：
        · toVueSFC(node)
        · toReact(node)
        · toUniApp(node)
        · toHTML(node)
```
- **混合策略**：结构/样式用**确定性代码生成器**（精准、可控），文案/变量命名/组件拆分建议用 **AI 增强**。
- 例：Node → Vue SFC 的 template 部分是确定性生成的；但"哪些节点该抽成子组件""变量怎么命名"交给 AI。

### 5.3 Provider 抽象接口
```ts
interface AIProvider {
  name: string
  generateFromImage(img: Buffer, opts): Promise<Node[]>      // 图生页面
  editByPrompt(tree: Node, prompt: string): Promise<Diff>     // 自然语言改
  convertFramework(tree: Node, target: 'vue'|'react'|'uni'|'html'): Promise<CodeFile[]>
  chat(messages): Promise<string>                             // 通用对话
}
```
适配器：`GLMProvider`（智谱）/ `OpenAIProvider` / `ClaudeProvider`（预留）。

## 7. 路线图（建议分期）

| 阶段 | 目标 | 产出 |
|---|---|---|
| **P0 地基**（先做） | Electron + Vue 工程、Node 树数据模型、画布渲染内核、基础拖拽选中 | 能拖几个组件到画布显示 |
| **P1 编辑器可用** | 属性面板、图层面板、样式编辑、撤销重做、设计令牌、内置组件库 ~15 个 | 一个能真正搭页面的编辑器 |
| **P2 导出** | HTML 单文件 / 分离文件导出、代码视图、本地预览服务器 | 产出可分享的 HTML |
| **P3 AI 图生** | Provider 抽象层、GLM-4V 接入、图→Node 树、加载进画布 | 截图变可编辑页面 |
| **P4 AI 微调** | 自然语言改页面、框选局部改、AI 改动可撤销 | 边说边改 |
| **P5 框架转译** | Vue3/React/uni/原生 四套生成器 + AI 增强命名/拆分 | 出工程化代码 |
| **P6 打磨** | 多页面/路由、模板库、版本快照、响应式断点、安装包打包 | 可发布产品 |

## 8. 待确认 / 风险点

1. **编辑器内核自研 vs 套用开源**：我倾向自研（为 AI/转译服务），但工作量更大。若想快速出 Demo，可先用 GrapesJS 顶 P0-P2，后期再换。**需你确认。**
2. **AI 图生准确率**：复杂设计稿一次成图难保证；建议首期做"单区块/单卡片"级识别，整页靠拼装。
3. **uni-app 转译深度**：完整工程结构（pages.json/manifest）较重，首期可只生成"页面级代码片段"，不生成完整工程脚手架。
4. **多模型密钥管理**：用 Electron safeStorage 加密存本地，首次需引导用户填 Key。
