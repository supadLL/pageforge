## 2. PR 切分原则

- 每个步骤文档对应一个主 PR。
- 如果某一步过大，优先拆成“基础能力”和“体验补充”两个 PR。
- PR 只做当前步骤要求的能力，不顺手扩展下一步功能。
- 每个 PR 完成后，必须满足该文档中的验收标准和测试建议。
- 文档中出现的“预留接口”不要求在当前 PR 做完整实现，只需能被后续步骤接上。

## 3. 文档模板

每个步骤文档都按以下结构编写：

- 范围
- 不做什么
- 前置依赖
- 实现要点
- 验收标准
- 测试建议
- PR Checklist

## 5. 统一约束

- 所有步骤都默认围绕 `Project / Page / Node / DesignTokens` 展开。
- 所有 UI 和导出能力都必须从 Node Tree 派生。
- 所有修改都走 command 或等价的统一更新入口。
- 所有新增字段必须写清楚默认值和兼容方式。
